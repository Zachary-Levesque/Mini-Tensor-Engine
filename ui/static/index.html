Project Summary

Mini Tensor Engine

This project is a custom C++ machine learning inference engine built from scratch. It runs a neural network, checks that the C++ output matches Python, and compares different matrix multiplication implementations to study performance.

Goal

What is this project trying to prove?

The goal is to show how machine learning inference works at a low level. Instead of using a large framework to hide the details, this project implements tensors, matrix multiplication, neural-network layers, validation, and benchmarking directly in C++.

Why it matters

Why is this relevant?

This is both a systems project and an ML infrastructure project. It demonstrates memory layout thinking, CPU performance work, correctness checking, and clean engineering design. It is useful for understanding how real inference runtimes are built.

Simple Explanation

How should you think about this project?

A neural network takes numbers in, does several math operations using learned weights, and produces numbers out. This engine is the software that performs that math. The important idea is that the exact same model can run at very different speeds depending on how tensors are stored and how matrix multiplication is implemented.

What is a backend?

A backend is the internal implementation the engine uses for matrix multiplication. The math result should stay the same, but the speed can change depending on how the computation is organized.

  • `naive`: the simplest version, easiest to understand, slowest
  • `transpose_rhs`: reorganizes memory access to be more cache-friendly
  • `threaded_transpose_rhs`: same cache-friendly idea, but split across multiple threads

What is a thread?

A thread is one worker that the CPU can use to do part of the job. More threads can make large computations faster by dividing the work, but they also add overhead. That is why more threads are helpful for big workloads and not always useful for tiny ones.

In this section, `Threads` only matters for the threaded backend. If you choose `naive` or `transpose_rhs`, the engine still runs on one thread.

What does this benchmark do?

              

                It measures how long the engine takes to run matrix multiplication and full model
                inference. Lower time means faster execution.
              

              
What do these settings mean?

              
    
                
  • `Iterations`: how many measured runs to perform. More iterations usually give a more stable average.
    • 
                
  • `Warmup`: how many runs to do first without timing them. This helps avoid measuring startup effects.
    • 
                
  • `Threads`: which thread counts to test for the threaded backend, for example `1,2,4`.
    • 
              

              

                A good way to think about it: the benchmark is asking, "How fast is this
                engine under different implementations and different levels of parallel work?"

MatMul Results

                

                  These results compare raw matrix multiplication performance. This is the lowest
                  level benchmark in the project.
                

              

              

            

            

              
What does a shape like `32x64x32` mean?

              

                It means the engine is multiplying a matrix with shape 32 x 64
                by a matrix with shape 64 x 32. The shared middle dimension is
                64, and the result has shape 32 x 32.
              

              
How do you read the result?

              

                Each row is one implementation. The number on the right is the average runtime.
                Lower is better. If one backend has a much smaller number, that backend is faster
                for that matrix size.
              

              

                This section helps answer: "Which matrix multiplication implementation is
                faster by itself?"

Model Results

                

                  These results measure full inference workloads, not just isolated matrix

Why is this section more important than the raw matmul section?

              

                Because a matrix multiplication trick can look fast in isolation but still fail to
                help the full model if the overhead is too high. This section measures the entire
                inference path from input to output.
              

              
What do names like `batch64_hidden256` mean?

              
    
                
  • `batch64`: 64 examples are processed at once
    • 
                
  • `hidden256`: the hidden layer width is 256
    • 
              

              

                Larger cases are more realistic for performance testing. They are also where
                threading tends to matter more.
              

              

                This section helps answer: "Does this optimization make the real model run
                faster?"
